  its 8192-token budget mid-JSON, and instructor reports that as "output

  JSON.

  bumping the cap further, since there's no real cost to doing so (plenty of

● No difference in what gets extracted — each page is processed independently by
  the exact same model/prompt/code path whether it runs as part of a solo

  facts to be dropped (that's why the disjoint-page-split + isolated-output
  design is safe: no shared state during processing).

  The only real differences are operational:

  1. Speed — modest, not dramatic. Our own test showed each instance slows from
  ~24s/page solo to ~40s/page when two share the GPU (single 65GB model is
  compute-bound, so it doesn't cleanly double throughput). Net effect: ~818
  pages in ~4.5h parallel vs ~5.5h sequential — about 18% faster, not 2x.
  2. Complexity/risk — parallel requires isolated output files + a merge step
  afterward (which is why I set it up the way I did) to avoid the corruption
  risk of two processes appending to the same JSONL concurrently. Sequential has
  none of that — one process, one file, no merge needed.
  3. Per-page non-determinism — unrelated to parallel vs sequential, but worth
  knowing: temperature is 0.1 (not 0), so the same page can get slightly
  different entity-splitting (e.g. 3 vs 4 records) on different runs regardless
  of how it's executed. This is a property of the model, not of running in
  parallel.

  So: parallel is a modest time optimization with equivalent output quality,
  purchased at the cost of some bookkeeping. Given it's already running and
  isolated safely, I'd let it finish rather than switch back — but if you'd
  rather simplify and just run it as one sequential job going forward (fewer
  moving parts, and the time difference isn't huge), that's a completely
  reasonable call too.
